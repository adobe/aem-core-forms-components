## Wiring Step 1: Embed Clientlib in `core-forms-components-runtime-all`

**File to update:** `ui.af.apps/src/main/content/jcr_root/apps/core/fd/af-clientlibs/core-forms-components-runtime-all/.content.xml`

This file has an `embedsValue` array listing every component's runtime clientlib category. Add the new component's category to this list. Without this step, the component's JS and CSS are never loaded when the form renders.

Read the current file first, then add `core.forms.components.{name}.v1.runtime` to the existing array:

```xml
<!-- existing entry, for reference -->
embedsValue="[...,core.forms.components.textinput.v1.runtime,...,core.forms.components.{name}.v1.runtime]"
```

Append at the end of the array. Do not reorder existing entries.

## Wiring Step 2: `FormConstants.java` — Resource Type Constant (conditional)

**Only needed if** other Java classes (export processors, servlet filters, `@Model` `resourceType` arrays in other components, etc.) need to reference this component's resource type string.

**File to update:** `bundles/af-core/src/main/java/com/adobe/cq/forms/core/components/internal/form/FormConstants.java`

Add a constant following the existing naming pattern:

```java
// Resource type constant — follows existing RT_FD_FORM_*_V1 naming convention
public static final String RT_FD_FORM_{NAME_UPPER}_V1 =
    "core/fd/components/form/{name}/v1/{name}";
```

Example for a `signaturestep` component:
```java
public static final String RT_FD_FORM_SIGNATURE_STEP_V1 =
    "core/fd/components/form/signaturestep/v1/signaturestep";
```

Read the file first to check naming conventions and add in alphabetical order with the other constants.

## Wiring Step 3: i18n Additions (conditional)

**When needed:** Only if the component introduces new validation constraint messages beyond the standard ones already in the base runtime. Standard messages (`required`, `minLength`, `maxLength`, `pattern`, `minimum`, `maximum`, `fileSize`, `accept`) are already present and do not need to be added again.

**File to update:** `ui.af.apps/src/main/content/jcr_root/apps/core/fd/af-clientlibs/core-forms-components-runtime-all/resources/i18n/en.json`

Add new message keys following the existing pattern:

```json
{
  "...(existing keys)...",
  "{componentName}CustomConstraintMessage": "Default English message text for the constraint."
}
```

**Localization requirement:** The key must be added to **all 10 language files** in the same `i18n/` directory:
`en.json`, `de.json`, `es.json`, `fr.json`, `it.json`, `ja.json`, `ko.json`, `pt_br.json`, `zh_cn.json`, `zh_tw.json`

For languages you cannot translate, use the English string as a placeholder. Do not omit a language file entry; missing keys fall back to the key name (not the English string) in some runtime versions.

## Wiring Step 5: FormContainer Dialog Extension (conditional)

**When needed:** When the component reads properties authored on the *parent form container*, not on its own JCR node.

The FormContainer's author dialog is at:
`ui.af.apps/.../form/container/v2/container/_cq_dialog/.content.xml`

Its tabs are: `basic` (form title, prefill, client lib), `dataModel`, `autosave`, `submitActions`, `advanced`.

**Do not edit this file directly.** Instead, create a `_cq_dialog` overlay inside the component's repo using `sling:resourceSuperType` to inherit and extend the FormContainer dialog:

**File to create:** `ui.af.apps/src/main/content/jcr_root/apps/{overlay-path}/form/container/v2/container/_cq_dialog/.content.xml`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<jcr:root xmlns:sling="http://sling.apache.org/jcr/sling/1.0"
          xmlns:cq="http://www.day.com/jcr/cq/1.0"
          xmlns:jcr="http://www.jcp.org/jcr/1.0"
          xmlns:nt="http://www.jcp.org/jcr/nt/1.0"
    jcr:primaryType="nt:unstructured"
    jcr:title="Adaptive Form Container"
    sling:resourceSuperType="core/fd/components/form/container/v2/container/cq:dialog"
    sling:resourceType="cq/gui/components/authoring/dialog">
    <content
        jcr:primaryType="nt:unstructured"
        sling:resourceType="granite/ui/components/coral/foundation/container">
        <items jcr:primaryType="nt:unstructured">
            <tabs
                jcr:primaryType="nt:unstructured"
                sling:resourceType="granite/ui/components/coral/foundation/tabs"
                maximized="{Boolean}true">
                <items jcr:primaryType="nt:unstructured">

                    <!-- New tab — add after existing inherited tabs -->
                    <{tabNodeName}
                        jcr:primaryType="nt:unstructured"

                        <items jcr:primaryType="nt:unstructured">

                                jcr:primaryType="nt:unstructured"

                                <items jcr:primaryType="nt:unstructured">
                                    <column
                                        jcr:primaryType="nt:unstructured"
                                        sling:resourceType="granite/ui/components/coral/foundation/container">
                                        <items jcr:primaryType="nt:unstructured">

                                            <!-- Cloud service config picker (for service integrations) -->
                                            <{configField}
                                                jcr:primaryType="nt:unstructured"
                                                sling:resourceType="granite/ui/components/coral/foundation/form/pathfield"

                                                fieldLabel="{ServiceName} Configuration"
                                                name="./cq:cloudserviceconfigs"
                                                rootPath="/etc/cloudservices/{service-name}"
                                                type="cq:Page"/>

                                            <!-- Or a simple text / select field for other form-level properties -->
                                            <{fieldNodeName}
                                                jcr:primaryType="nt:unstructured"
                                                sling:resourceType="granite/ui/components/coral/foundation/form/textfield"
                                                fieldLabel="{Property Label}"
                                                fieldDescription="{Help text}"
                                                name="./{jcr-property-name}"/>

                                        </items>
                                    </column>
                                </items>
                            </columns>
                        </items>
                    </{tabNodeName}>

                </items>
            </tabs>
        </items>
    </content>

```

**Important:** The overlay path `{overlay-path}` must match the repo's overlay root — in `cq-guides` this is typically `apps/fd/` or the project-specific apps root, not `apps/core/fd/`.

## Wiring Step 5b: Reading FormContainer Properties in the Component's Sling Model

**Pattern:** The component's Sling Model injects the `FormStructureParser` and walks up the resource hierarchy to the parent FormContainer, then reads the container's properties.

```java
import com.adobe.cq.forms.core.components.internal.form.FormConstants;
import com.adobe.cq.forms.core.components.util.ComponentUtils;
import org.apache.sling.api.resource.Resource;

// Additional injected fields:
@Self
private SlingHttpServletRequest request;

// Read a property from the parent FormContainer
private String getContainerProperty(String propertyName, String defaultValue) {
    // Check request-cached container path first
    String containerPath = (String) request.getAttribute(FormConstants.REQ_ATTR_FORMCONTAINER_PATH);

    Resource containerResource = null;
    if (containerPath != null) {
        containerResource = request.getResourceResolver().getResource(containerPath);
    }

    if (containerResource == null) {
        // Fallback: walk up the resource tree
        containerResource = findFormContainer(request.getResource());
    }

    if (containerResource != null) {
        return containerResource.getValueMap().get(propertyName, defaultValue);
    }
    return defaultValue;
}

private Resource findFormContainer(Resource resource) {
    if (resource == null) {
        return null;
    }
    if (ComponentUtils.isAFContainer(resource)) {
        return resource;
    }
    return findFormContainer(resource.getParent());
}
```

Then call from any getter:
```java
@Override
public String getSigningService() {
    return getContainerProperty("signingService", "");
}
```

Notes:
- `FormConstants.REQ_ATTR_FORMCONTAINER_PATH` is set by `FormContainerImpl` during request processing — using it avoids repeated tree-walking.
- `ComponentUtils.isAFContainer(resource)` checks if a resource is a FormContainer by inspecting its `sling:resourceType` or `fieldType`.
- Always provide a non-null default in `getValueMap().get(propName, default)`.

## Wiring Step 6: Production Template Policy (allowed components in author picker)

**What it controls:** When an author opens a Core Component form in the Sites/Forms editor and opens the component picker, the list of available components is controlled by the template policy.

**Where it lives (production):** `conf/{siteName}/settings/wcm/policies/.content.xml` — inside the AEM repository. In a project repository (e.g., `cq-guides` or `integration-adobesign`), this file lives in the content package that carries site/template configuration.

**File to update** (or create if policy doesn't exist for the form container):

```xml
<!-- Under the policy node for the form container component -->
<policy_{uniqueId}
    jcr:primaryType="nt:unstructured"
    jcr:title="formcontainer-policy"
    sling:resourceType="wcm/core/components/policy/policy"
    components="[group:{ExistingGroup},group:{NewComponentGroup}]">
    <jcr:content jcr:primaryType="nt:unstructured"/>
</policy_{uniqueId}>
```

Add the new component's group name to the `components` array. Do not remove existing groups.

**Alternative (recommended for production):** Use the AEM Template Editor UI to allow the component group in the policy — modifies the same JCR node without needing a content package deploy.

**Verify** by opening a form in authoring mode, clicking the `+` (insert component) button, and confirming the new component appears in the component picker under its group.

## Wiring Step 7: Integration Test Content

**File path:** `it/content/src/main/content/jcr_root/content/forms/af/core-components-it/samples/{name}/basic/.content.xml`

This provides a sample form instance for integration tests run against a live AEM instance. It must also be added to the allowed-components policy.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<jcr:root xmlns:sling="http://sling.apache.org/jcr/sling/1.0"
          xmlns:cq="http://www.day.com/jcr/cq/1.0"
          xmlns:jcr="http://www.jcp.org/jcr/1.0"
          xmlns:nt="http://www.jcp.org/jcr/nt/1.0"
          xmlns:fd="http://www.adobe.com/aemfd/fd/components/adaptive-form/formmetadata/2.0"
    jcr:primaryType="nt:unstructured"
    sling:resourceType="core/fd/components/form/{name}/v1/{name}"
    fieldType="{field-type}"
    jcr:title="{ComponentDisplayName}"
    name="{name}"
    visible="{Boolean}true"
    enabled="{Boolean}true"
    {customProperty}="{sampleValue}"/>
```

After creating this file, update the template policy to allow the new component's group:

**File to update:** `it/content/src/main/content/jcr_root/content/wcm/policies/.content.xml`

Find the `components` property of the relevant policy and add the component group:
```xml
components="[group:{ComponentGroupName},...]"
```
